# Product Overview

## 목적

이 문서는 이 프로젝트가 무엇을 만드는지, 어떤 원칙을 절대 지켜야 하는지, 기능을 어떤 기준으로 나눠야 하는지를 AI가 빠르게 이해하도록 돕기 위한 제품 기준 문서다.

이 문서는 사람 소개용 README가 아니다.
구현, 구조 설계, 기능 분리, 권한 처리, 익명성 정책을 판단할 때 우선 참고하는 기준이다.

---

## 프로젝트 한 줄 정의

이 프로젝트는 교육 현장에서 구성원들이 서로의 실제 신원을 직접 알지 못한 상태에서, 익명 닉네임과 구조화된 협업 흐름을 기반으로 공정하게 소통하고 평가받을 수 있도록 돕는 익명 협업 교육 플랫폼이다.

---

## 문제의식

현재의 조별 과제, 토론, 협업 수업은 과제 해결보다 인간관계의 역학에 더 큰 영향을 받는다.

대표 문제는 아래와 같다.

1. 친구가 없거나 소외된 학생은 시작 전부터 심리적으로 위축된다.
2. 목소리가 큰 학생이 대화를 주도하면 다양한 아이디어가 묻힌다.
3. 협업의 질보다 친분, 서열, 분위기가 더 강하게 작용한다.
4. 동료 평가는 쉽게 인기투표나 감정 평가로 변질된다.
5. 무임승차와 언어 폭력 같은 부작용을 공정하게 통제하기 어렵다.

이 프로젝트는 위 문제를 기술적으로 완화하는 것을 목표로 한다.

---

## 핵심 컨셉

핵심 컨셉은 아래 한 줄로 요약할 수 있다.

`실제 신원이 아니라 익명 닉네임, 데이터, 논리, 기여도 기반으로 협업하는 교육 환경`

즉, 이 프로젝트는 단순한 수업 관리 시스템이 아니다.
핵심은 아래에 있다.

1. 완전 익명에 가까운 협업 경험
2. 역할 기반 기능 분리
3. AI 기반 닉네임 생성
4. AI 기반 모둠 자동 구성
5. AI 기반 기여도 분석 및 공정 평가 보조

---

## 역할 정의

### student

수강생 역할이다.
익명 닉네임을 중심으로 수업에 참여하고, 모둠 활동과 과제 제출, 공지 확인, 협업 대화에 참여한다.

### teacher

교강사 역할이다.
수업을 만들고, 수업 코드를 생성하고, 모둠을 구성하고, 공지사항을 작성하고, 모둠별 제출 상태와 기여도 결과를 확인한다.

중요:

- 로그인 시 사용자 역할은 `student`, `teacher` 중 하나다.
- 시스템은 역할에 따라 노출 기능과 접근 기능이 달라진다.
- 학생 화면과 교강사 화면은 완전히 달라질 수 있다.

---

## 익명성 원칙

이 프로젝트에서 가장 중요한 제품 규칙 중 하나다.

1. 학생은 다른 학생의 본명을 보지 않는다.
2. 학생은 익명 닉네임 중심으로 협업한다.
3. 본명은 교강사에게만 노출될 수 있다.
4. 닉네임은 협업 화면에서 기본 식별자다.
5. 본명과 닉네임은 같은 개념이 아니다.
6. 익명성은 부가 기능이 아니라 제품의 핵심 가치다.

중요:

- 닉네임 관련 기능은 단순 프로필 기능이 아니라 핵심 도메인이다.
- 본명 노출 정책은 권한 처리에서 매우 중요하다.

---

## 회원가입 / 인증 원칙

### 회원가입

회원가입은 로컬 회원가입을 기준으로 한다.

포함 요소:

1. 이메일 기반 가입
2. 이메일 인증
3. 역할 선택
4. 닉네임 자동 생성
5. 닉네임 중복 처리
6. AI 기반 닉네임 자동 생성
7. 닉네임 다시 설정
8. 본명 저장
9. 성격, 취향, MBTI 등 설문 수집

중요:

- 설문 수집은 `student` 역할에만 적용한다.
- 본명은 저장되지만 기본 협업 식별자는 닉네임이다.
- 본명은 교강사에게만 보이게 해야 한다.

### 로그인

로그인은 JWT + RFR 구조를 사용한다.

포함 요소:

1. access token 발급
2. refresh token 발급
3. refresh token rotation
4. logout 시 refresh token 폐기
5. 역할 기반 로그인 처리

중요:

- 역할은 사용자 계정 상태와 연결된다.
- 역할 변경 정책은 추후 명확히 설계해야 하지만, 현재 문맥상 로그인 시 `student / teacher` 사용 경험이 명확히 분리되는 구조를 의미한다.

## AI가 관여하는 기능

이 프로젝트에서 AI는 선택적 장식 기능이 아니라 핵심 가치 실현 도구다.

주요 AI 기능은 아래와 같다.

1. 닉네임 자동 생성
2. 닉네임 재생성
3. 학생 설문 성향 분석
4. AI 기반 모둠 자동 편성
5. 비슷한 성향 / 상반된 성향 기준 추천
6. 기여도 결과 분석
7. 평가 시각화에 필요한 판단 보조

중요:

- AI 기능은 익명성과 공정성 강화에 기여해야 한다.
- AI 기능이 본명 노출이나 익명성 훼손 방향으로 설계되면 안 된다.

## 도메인 구조 기준

이 프로젝트는 화면이 학생/교강사로 크게 다를 수 있다.
하지만 백엔드 구조는 무조건 `student`, `teacher` 폴더부터 나누는 방식으로 시작하지 않는다.

기본 원칙:

1. 먼저 기능 중심으로 본다.
2. 역할 차이는 그 기능 안에서 분기한다.
3. 정말 역할별 기능 묶음이 거의 겹치지 않을 때만 상위 역할 폴더 분리를 검토한다.

예:

- `auth`
- `class`
- `group`
- `assignment`
- `notice`
- `contribution`
- `survey`
- `profile`

중요:

- 프론트 화면이 다르다는 이유만으로 백엔드 최상위를 역할별로 분리하지 않는다.
- 권한 분기와 화면 분기는 다른 문제다.

## 현재 auth 구조 기준

현재 인증 구조는 아래 기준을 따른다.

```text
src/auth/
  login/
    dto/
    token/
    refresh-token/
    login.controller.ts
    login.service.ts
  logout/
    dto/
    logout.controller.ts
    logout.service.ts
  signup/
    dto/
    entities/
    password.service.ts
    signup.controller.ts
    signup.service.ts
```

해석:

1. `login/token`은 로그인 흐름 전용 토큰 책임이다.
2. `login/refresh-token`은 로그인 세션 연장 책임이다.
3. `logout`은 별도 기능이다.
4. `signup`은 가입 흐름 책임이다.

## 절대 어기면 안 되는 제품 규칙

아래 규칙은 구조 제안이나 기능 구현 시 항상 우선한다.

1. 학생 협업 화면의 기본 식별자는 본명이 아니라 닉네임이다.
2. 학생에게 다른 학생의 본명을 노출하면 안 된다.
3. 교강사는 본명을 볼 수 있다.
4. AI 기능은 익명성과 공정성을 강화하는 방향이어야 한다.
5. 로그인/재발급/로그아웃은 JWT + RFR 기준을 따라야 한다.
6. refresh token은 서버 기준으로 관리되어야 한다.
7. 기능 경계가 다르면 폴더를 나눈다.
8. 역할 차이만으로 무조건 상위 폴더를 나누지 않는다.
9. 필요 없는 폴더는 만들지 않는다.
10. 제품의 핵심은 단순 수업 관리가 아니라 익명 협업과 공정한 평가다.

## AI가 구조 판단 시 우선 질문할 것

이 프로젝트에서 AI는 기능을 만들기 전에 아래를 먼저 확인해야 한다.

1. 이 기능은 익명성 정책과 충돌하지 않는가?
2. 이 기능은 학생용인가, 교강사용인가, 공통인가?
3. 이 기능은 기존 도메인 어디에 속하는가?
4. 이 기능은 본명 기반인가, 닉네임 기반인가?
5. AI가 관여하는 기능인가?
6. 권한 차이인지, 도메인 차이인지 구분했는가?
7. 구조를 나누는 이유가 진짜 기능 경계 때문인가?
