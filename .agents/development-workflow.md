# Development Workflow

## 목적
이 문서는 이 프로젝트에서 기능을 설계하고 구현할 때 어떤 순서로 생각하고, 어떤 기준으로 질문하고, 어떻게 구조를 잡아야 하는지를 정의한다.

목표:

1. 애매한 상태로 바로 구현하지 않는다.
2. 레이어 책임을 먼저 구분한다.
3. 도메인과 하위 기능을 먼저 정리한다.
4. 주니어 개발자도 이해 가능한 방식으로 설명한다.
5. 구현보다 설계를 먼저 분명하게 한다.

---

## 작업 시작 전 기본 원칙
이 프로젝트에서는 코드를 바로 쓰지 않는다.

항상 아래 순서로 먼저 확인한다.

1. 지금 만들 기능이 어느 도메인에 속하는가?
2. 그 도메인 안에서 어떤 하위 기능인가?
3. 어느 레이어의 책임인가?
4. 요구사항이 충분히 구체적인가?
5. 기억이 불명확한 내용은 없는가?
6. 질문이 필요한 상태는 아닌가?

---

## 반드시 질문해야 하는 상황
아래 상황이면 추측하지 말고 먼저 질문한다.

1. 요구사항이 애매할 때
2. 도메인 경계가 불명확할 때
3. 하위 기능 분리가 애매할 때
4. 레이어 책임이 헷갈릴 때
5. API 응답 형식이 정해지지 않았을 때
6. 예외 처리 정책이 정해지지 않았을 때
7. DB 모델링 기준이 정해지지 않았을 때
8. 테스트 범위가 정해지지 않았을 때
9. 네이밍 후보가 여러 개고 의미 차이가 있을 때
10. 이전 대화 내용을 확실히 기억하지 못할 때
11. 비즈니스 규칙이 누락되었을 가능성이 있을 때

---

## 질문 형식
질문은 막연하게 하지 않는다.
항상 아래 구조를 따른다.

1. 현재 헷갈리는 지점
2. 가능한 선택지
3. 각 선택지의 차이
4. 내가 결정해야 하는 핵심 포인트

예시:

- 현재 헷갈리는 지점
  주문 취소를 `order/cancel`로 분리할지 `order/status` 안에 둘지 애매함

- 가능한 선택지
  1. `order/cancel` 하위 기능으로 분리
  2. `order/status` 안에서 상태 변경 유스케이스로 처리

- 각 선택지의 차이
  1번은 취소 정책이 커질 때 유리함
  2번은 단순 상태 변경이면 구조가 덜 복잡함

- 내가 결정해야 하는 핵심 포인트
  주문 취소가 단순 상태 변경인지, 별도 환불/검증 정책까지 포함하는지

## 기능 구현 전 사고 순서
기능을 만들기 전에 항상 아래 순서로 생각한다.

### 1. 도메인 결정
이 기능이 어느 도메인에 속하는지 먼저 판단한다.

예:
- 로그인 → `user/auth`
- 프로필 수정 → `user/profile`
- 결제 승인 → `order/payment`
- 재고 차감 → `product/inventory`

### 2. 하위 기능 결정
도메인 안에서 어떤 하위 기능인지 정한다.

예:
- `user/auth`
  - 로그인
  - 로그아웃
  - 토큰 재발급

- `order/payment`
  - 결제 요청
  - 결제 승인
  - 결제 실패 처리

### 3. 레이어 결정
그 기능이 어느 레이어의 책임인지 먼저 정한다.

예:
- HTTP 입력 검증 → `presentation`
- 유스케이스 실행 → `application`
- 비즈니스 규칙 판단 → `domain`
- TypeORM 저장 구현 → `infrastructure`

### 4. 파일 위치 결정
코드를 쓰기 전에 어떤 파일이 어디 있어야 하는지 먼저 정리한다.

예:

```text
src/
  domains/
    user/
      auth/
        presentation/
          controllers/
            login.controller.ts
          dtos/
            login.request.dto.ts
            login.response.dto.ts
        application/
          use-cases/
            login-user.use-case.ts
        domain/
          repositories/
            user.repository.ts
        infrastructure/
          repositories/
            typeorm-user.repository.ts
```

### 5. 책임 분리 확인
한 파일에 여러 책임이 섞이지 않는지 반드시 확인한다.

예:
- controller가 토큰 생성까지 하면 안 된다.
- use-case가 TypeORM repository를 직접 `new` 하면 안 된다.
- domain entity에 TypeORM decorator를 넣으면 안 된다.

## 구현 전 체크리스트
코드 작성 전에 아래를 스스로 점검한다.

1. 이 기능은 어느 도메인 소속인가?
2. 기존 하위 기능에 속하는가?
3. 새 하위 기능이 필요한가?
4. 어느 레이어 책임인가?
5. controller에 비즈니스 로직이 들어가려는가?
6. application이 ORM 세부사항을 알고 있는가?
7. domain이 외부 라이브러리에 의존하는가?
8. repository interface와 구현체가 분리되어 있는가?
9. 이름이 역할을 정확히 드러내는가?
10. 주니어 개발자가 읽고 이해 가능한 구조인가?

---

## 구현 중 체크리스트
구현하면서 아래를 계속 확인한다.

1. 설명 없이 코드만 쓰고 있지 않은가?
2. service 하나에 책임을 몰아넣고 있지 않은가?
3. DTO와 Entity를 혼동하지 않았는가?
4. domain 규칙이 infrastructure에 새어 나가지 않았는가?
5. DB 세부사항이 domain으로 들어오지 않았는가?
6. use-case가 너무 커지지 않았는가?
7. 파일 하나가 너무 많은 일을 하지 않는가?

---

## 구현 후 리뷰 체크리스트
구현이 끝난 뒤 아래를 점검한다.

1. 파일 위치가 맞는가?
2. 이름이 책임을 드러내는가?
3. 레이어 침범이 없는가?
4. 도메인 경계가 유지되는가?
5. 공통화가 과하지 않은가?
6. 주니어 개발자가 따라올 수 있는가?
7. 변경이 생겼을 때 수정 포인트를 쉽게 찾을 수 있는가?
8. 테스트 가능한 구조인가?

---

## 자주 하는 실수

### 1. service 하나에 몰아넣기
가장 흔한 실수다.

예:
- `UserService` 안에 로그인
- 회원가입
- 프로필 수정
- 권한 확인
- 토큰 발급
- 비밀번호 변경

이렇게 전부 넣어버리면 책임이 섞인다.

### 2. controller에 비즈니스 로직 넣기
controller는 입구다.
비즈니스 로직 처리 장소가 아니다.

### 3. domain과 DB 모델을 같은 것으로 보기
TypeORM entity는 저장 모델이다.
domain entity는 비즈니스 모델이다.

같다고 생각하면 레이어가 무너진다.

### 4. 애매한 상태에서 일단 구현하기
질문해야 할 상황인데 추측으로 구현하면 나중에 구조가 틀어질 가능성이 크다.

### 5. 설명 없이 결과만 던지기
이 프로젝트에서는 작동하는 코드만으로 충분하지 않다.
왜 이 구조인지 설명 가능해야 한다.

## 금지 행동
다음 행동은 하지 않는다.

1. 요구사항이 애매한데 추측으로 진행하기
2. 기억이 불명확한데 아는 척하고 진행하기
3. 구조 설명 없이 코드부터 작성하기
4. controller에 로직 몰아넣기
5. application에서 DB 세부 구현 직접 처리하기
6. domain에 TypeORM 같은 기술 의존 넣기
7. 이름이 애매한 파일 만들기
8. 공통화부터 먼저 하려 하기

## 좋은 응답의 기준
좋은 응답은 아래를 만족해야 한다.

1. 구체적이다.
2. 단계별이다.
3. 추측하지 않는다.
4. 레이어를 분리해서 설명한다.
5. 도메인과 하위 기능을 먼저 본다.
6. 파일 구조를 실제 트리 형태로 보여준다.
7. 왜 그렇게 하는지 설명한다.
8. 주니어 기준으로 이해 가능하다.
